# AGENTS.md

이 문서는 이 저장소에서 작업하는 코딩 에이전트가 따라야 하는 **리포지토리 전역 규칙**을 정의합니다.

## 미션

이 저장소는 **프로덕트 매니저를 위한 데스크형 멀티 에이전트 AI 워크스페이스**를 구축하기 위한 프로젝트입니다.

이 시스템은 일반적인 단일 챗봇이 아닙니다. 다음 특성을 갖는 구조화된 제품입니다.

* 여러 개의 가시적인 에이전트
* 공유 메모리
* 역할 기반 추론
* 제어된 선제 발화
* 구조화된 에이전트 간 협업

코드를 수정하거나 추가할 때는 반드시 이 제품 방향성을 유지해야 합니다.

## 제품 가드레일

구현 시 항상 아래 원칙을 지키세요.

1. **단일 거대 어시스턴트가 아니라, 역할이 분리된 여러 에이전트**
2. **기억은 공유하지만, 각 에이전트의 관점은 다름**
3. **사용자에게는 항상 하나의 최종 화자만 응답**
4. **에이전트 간 통신은 반드시 구조화되어야 함**
5. **선제 발화는 제한적이고 우선순위 기반이어야 함**
6. **외부 도구는 source of truth로 취급**
7. **MVP는 read-first로 시작**
8. **쓰기 동작은 승인 기반이어야 함**
9. **초기에 과도한 자동화를 하지 말 것**
10. **신기함보다 PM 워크플로우의 명확성을 우선할 것**

## 현재 에이전트 역할

### 서비스 기획 및 아이디에이션 Agent

담당:

* 서비스별 맥락 이해
* 아이디어 저장
* 시장 조사
* 경쟁사 업데이트 추적
* 기능/제품 아이디에이션 지원

### 제품 운영 Agent

담당:

* 프로젝트 추적
* 마감일 모니터링
* blocker 감지
* 일일 운영 브리핑
* 실행 가능성 검토

### 개인 비서 Agent

담당:

* 개인 일정 파악
* 회의 준비 지원
* 개인 업무 정리
* 멘션/주의 대화 분류
* 일일 실행 보조

## 엔지니어링 우선순위

구현 시 우선순위는 아래와 같습니다.

1. 명확한 인터페이스
2. 컴포넌트 간 올바른 경계
3. mock 가능한 연동 구조
4. 추적 가능성 / 로그
5. 안전한 기본값
6. UX 및 표현 다듬기

외부 서비스 연동을 깊게 붙이기 전에, 내부 인터페이스와 경계를 먼저 안정화하세요.

## 저장소 선호사항

### 선호하는 접근 방식

* 아키텍처는 모듈형으로 유지
* 명시적인 타입과 계약을 선호
* 오케스트레이션과 비즈니스 규칙을 분리
* 메모리 모델과 툴 어댑터를 분리
* 외부 연동은 반드시 인터페이스 뒤에 둘 것
* mock 구현을 실제 연동으로 쉽게 교체할 수 있게 만들 것

### 피해야 할 것

* UI에 제품 로직 하드코딩
* 프로토콜 없이 에이전트끼리 임의로 호출
* 연동 로직과 추론 로직의 강한 결합
* 승인 없는 외부 쓰기 동작
* 감사(audit) 불가능한 과도하게 마법 같은 동작

## 예상 저장소 구조

장기적으로는 아래 구조를 지향합니다.

* `apps/desk-ui`: 사용자용 데스크 인터페이스
* `apps/orchestrator`: 요청 라우팅 및 응답 조정
* `packages/shared-memory`: 메모리 모델 및 메모리 서비스
* `packages/agent-protocol`: 에이전트 메시지 계약
* `packages/tool-connectors`: Slack / Asana / Calendar / Notion / Web 어댑터
* `docs/`: 제품 및 시스템 설계 문서

아직 이 구조가 전부 구현되어 있다고 가정하지 말고, 점진적으로 추가하세요.

## 런타임 동작 요구사항

런타임 로직을 구현할 때는 아래를 만족해야 합니다.

* 사용자는 특정 에이전트를 직접 선택할 수 있어야 함
* 한 에이전트는 다른 에이전트에게 정보 요청/검토 요청 가능
* 내부 협업은 request/response 기반이어야 함
* 사용자에게 보이는 최종 응답은 하나의 지정된 에이전트가 생성해야 함
* 시스템은 아래를 로그로 남길 수 있어야 함:

  * 무엇이 요청되었는가
  * 어떤 에이전트가 관여했는가
  * 어떤 도구가 사용되었는가
  * 어떤 confidence가 부여되었는가
  * 어떤 memory가 조회/수정되었는가

## Source-of-Truth 정책

아래 시스템은 각각의 사실 기준으로 취급합니다.

* Asana: 프로젝트/업무 상태
* Google Calendar: 일정 사실
* Notion: 개인 업무 기록
* Slack: 최종 진실이 아닌 signal source
* Web Search: 시장/경쟁사 맥락 정보

공유 메모리는 유용한 작업 기억을 저장해야 하지만, 외부 source of truth를 잘못 덮어쓰면 안 됩니다.

## MVP 제약

초기 MVP에서는 아래를 지원합니다.

* 외부 도구 읽기
* 구조화된 요약
* 일일 브리핑 로직
* 에이전트 간 내부 협업
* 서비스/아이디어/프로젝트/개인 업무용 공유 메모리

기본적으로 자동 외부 쓰기 기능은 구현하지 않습니다.

## 의사결정 원칙

다음 둘 사이에서 선택해야 한다면:

* 화려한 동작
* 제어 가능하고 감사 가능한 동작

항상 **제어 가능하고 감사 가능한 동작**을 선택하세요.
