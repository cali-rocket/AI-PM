# 제품 컨셉

## 개요

이 제품은 **프로덕트 매니저를 위한 데스크형 멀티 에이전트 워크스페이스**입니다.

사용자는 하나의 단일 어시스턴트와 대화하는 대신, 서로 다른 역할을 가진 여러 AI 에이전트와 하나의 작업 공간 안에서 상호작용합니다. 각 에이전트는 고유한 역할, 관점, 책임 범위를 가지며, 필요할 때 제한적으로 선제 개입할 수 있습니다.

사용자 경험은 다음과 같아야 합니다.

* 여러 명의 유능한 동료가 내 책상 위에 앉아 있음
* 나는 원하는 에이전트에게 직접 말을 걸 수 있음
* 에이전트들은 서로 조용히 상의할 수 있음
* 최종적으로는 한 에이전트가 명확하게 답변함

이 제품은 “여러 페르소나가 있는 챗봇”이 아니라, **PM 업무를 위한 역할 기반 AI 운영 환경**입니다.

## 핵심 비유

**“내 책상 위에 앉아 있는 AI 팀”**

사용자는 다음과 같은 구조를 경험합니다.

* 눈에 보이는 에이전트 목록 (dock/desk 형태)
* 역할별 대화 공간
* 에이전트별 inbox / 업데이트
* 문맥 있는 선제 알림
* 전략, 운영, 개인 실행을 연결하는 협업 경험

## 이 제품이 필요한 이유

PM의 일은 아래처럼 파편화되어 있습니다.

* 제품/서비스 사고
* 프로젝트 조율
* 이해관계자 커뮤니케이션
* 개인 일정 관리
* 후속 실행 추적

단일 챗 인터페이스는 편리하지만, 이 모든 맥락을 하나의 스레드에 섞어버리기 쉽습니다.

이 제품은 이를 해결하기 위해 다음과 같이 역할을 분리합니다.

* 제품/서비스 사고를 돕는 에이전트
* 전달/운영 상황을 추적하는 에이전트
* 개인 실행을 돕는 에이전트

이를 통해 더 자연스럽고, 더 제어 가능하며, 더 신뢰할 수 있는 상호작용을 제공합니다.

## 핵심 사용자 가치

이 제품은 사용자가 다음을 더 잘 하도록 도와야 합니다.

1. 더 명확하게 사고하기
2. 더 잘 기억하고 연결하기
3. 실행 상황을 더 안정적으로 추적하기
4. 리스크를 더 빨리 인지하기
5. 전략적 판단과 실제 운영 제약을 연결하기
6. 컨텍스트 스위칭 비용 줄이기

## 주요 사용 시나리오

### 1. 서비스 기획 및 아이디에이션

사용자는 서비스 기획 Agent와 기능 아이디어, 방향성, 경쟁사 움직임, 시장 정보 등을 논의합니다.

### 2. 제품 운영 파악

사용자는 제품 운영 Agent에게 프로젝트 상태, 마감일, blocker, 리소스 부담, 임박 리스크를 묻습니다.

### 3. 개인 실행 보조

사용자는 개인 비서 Agent에게 오늘 일정, 지금 해야 할 일, 후속 작업, 중요한 멘션 등을 묻습니다.

### 4. 에이전트 협업

사용자는 한 에이전트로 대화를 시작하지만, 해당 에이전트는 필요 시 다른 에이전트와 협업한 후 답변할 수 있습니다.

예:

* 아이디어 검토 시 운영 가능성 확인
* 우선순위 질문 시 캘린더 여유 시간 확인
* 회의 준비 시 관련 프로젝트 맥락 조회

## 상호작용 모델

### 직접 상호작용

사용자가 특정 에이전트를 명시적으로 선택해 대화를 시작합니다.

### 선제 상호작용

에이전트는 특정 조건이 충족될 때 제한적으로 먼저 사용자에게 말을 겁니다.

알림 수준:

* 긴급
* 중요
* 정보성

### 내부 협업

에이전트는 구조화된 프로토콜을 통해 서로에게 정보를 요청하거나 검토를 받을 수 있습니다.

## UX 원칙

* 각 에이전트는 분명한 전문성을 가져야 한다
* 불필요하게 자주 끼어들면 안 된다
* 워크스페이스는 복잡도를 줄여야 한다
* 최종 응답은 언제나 명확해야 한다
* 시스템은 혼란스럽지 않고 협업적으로 느껴져야 한다

## 이 제품의 차별점

이 제품은 단순히 다음과 같은 것이 아닙니다.

* 여러 도구를 붙인 래퍼
* 알림 모음판
* 페르소나만 나뉜 챗봇

이 제품은 다음과 같습니다.

* 역할 기반 AI 운영 환경
* PM 워크플로우 중심 설계
* 공유 메모리와 구조화된 협업 기반
* 장기적인 문맥 축적을 전제로 한 시스템

## 제품 원칙

1. 하나의 워크스페이스, 여러 명의 전문가
2. 기억은 공유하지만 판단은 역할별
3. 선제 발화는 유용해야지 시끄러우면 안 됨
4. 내부 협업은 결과를 더 좋아지게 해야 함
5. 자동화의 과시보다 통제와 신뢰가 중요함
6. 쓰기보다 읽기부터 시작
7. PM 워크플로우 명확성이 최우선

## 초기 제품 경계

첫 버전은 다음에 집중합니다.

* 이해
* 요약
* 우선순위화
* 리마인드
* 브리핑
* 맥락 기반 판단 지원

처음부터 완전 자율 실행 시스템으로 시작하지 않습니다.

## 장기 방향성

시간이 지나면 다음으로 확장할 수 있습니다.

* 승인 기반 액션 실행
* task 생성/수정 제안
* 메시지 초안 작성
* 크로스툴 follow-up 오케스트레이션
* 더 정교한 memory 정제/회수

하지만 기반은 항상 아래여야 합니다.

**명확한 역할, 안전한 동작, 구조화된 협업, 높은 신뢰**
