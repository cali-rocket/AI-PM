# MVP 범위

## 개요

MVP는 너무 이른 시점에 위험한 자동화를 시도하지 않고, 핵심 제품 경험을 검증하는 데 집중해야 합니다.

MVP의 목표는 아래를 증명하는 것입니다.

* 데스크형 멀티 에이전트 상호작용 모델이 실제로 유용한가
* 역할이 분리된 에이전트가 PM 워크플로우를 더 명확하게 만드는가
* 공유 메모리와 구조화된 협업이 더 나은 결과를 만드는가
* 제어된 선제 알림이 가치가 있는가
* 외부 도구 읽기 연동만으로도 실질적인 도움이 되는가

MVP는 **완전 자율 실행 시스템**을 목표로 하지 않습니다.

---

## MVP 목표

MVP는 아래 가설을 검증해야 합니다.

1. 사용자는 하나의 평평한 챗봇보다 역할이 분리된 에이전트와 대화할 때 더 큰 가치를 느낀다
2. 에이전트 간 협업은 PM 업무에 대한 응답 품질을 높인다
3. 공유 메모리는 대화 연속성과 회수(recall)에 도움을 준다
4. 도구 기반 읽기 접근만으로도 빠른 실용 가치를 만든다
5. 통제된 선제 알림은 시끄럽지 않으면서도 유용할 수 있다

---

## 포함 범위

### 1. 데스크형 멀티 에이전트 UX

* 가시적인 에이전트 모델
* 사용자가 특정 에이전트를 직접 선택 가능
* 각 에이전트별 역할 컨텍스트
* 에이전트별 상태 및 업데이트 표시 (초기엔 단순해도 됨)

### 2. 3개의 핵심 에이전트

* 서비스 기획 및 아이디에이션 Agent
* 제품 운영 Agent
* 개인 비서 Agent

### 3. 공유 메모리 (초기 버전)

아래에 대한 구조화된 memory 지원:

* services
* ideas
* project summaries
* personal task summaries
* important context notes

### 4. 구조화된 에이전트 협업

아래 유형의 제한된 내부 협업 프로토콜 지원:

* information requests
* feasibility checks
* schedule checks
* priority review
* context summary requests
* risk checks

### 5. 읽기 연동

아래 도구에 대한 read-only 또는 read-dominant 연동:

* Slack
* Asana
* Google Calendar
* Notion
* Web Search

### 6. 실용적인 사용자 플로우

최소한 아래 플로우를 지원:

* 새로운 제품 아이디어 논의 및 저장
* 어떤 프로젝트가 위험한지 질문
* 오늘 무엇을 해야 하는지 질문
* 매일 정오 운영 브리핑 수신
* 하루 시작 개인 브리핑 수신
* 회의 준비용 맥락 브리핑 받기
* 한 도메인의 질문을 다른 도메인 맥락과 연결하는 협업 응답

### 7. 제어된 선제 알림

아래 수준의 선제 업데이트 허용:

* urgent
* important
* informational

초기 동작은 보수적이어야 합니다.

---

## MVP에서 명시적으로 제외되는 것

### 1. 승인 없는 제3자 쓰기

다음 동작은 자동으로 하지 않습니다.

* Asana task 생성/수정
* Notion task 생성/수정
* Slack 메시지 전송
* Calendar 이벤트 변경

### 2. 완전 자율 운영

시스템은 다음을 하지 않습니다.

* 독립적으로 프로젝트 운영
* 완전 자율 오퍼레이터처럼 행동
* 사용자 검토 없이 의사결정 실행

### 3. 고도화된 메모리 지능

초기 런칭에서 다음은 필수 요구가 아닙니다.

* 매우 정교한 장기 memory ranking
* 완벽한 자동 중복 제거
* 완전 자동 memory pruning

### 4. 무거운 워크플로우 자동화

다음으로 시작하지 않습니다.

* 복잡한 다단계 자동 실행
* 여러 도구에 걸친 연쇄 액션
* 승인 없는 자동 follow-up 루프

### 5. 과도한 프로덕션 UI 폴리시

첫 버전은 아래가 필수는 아닙니다.

* 아주 정교한 시각적 인터랙션
* 애니메이션 에이전트 아바타
* 데스크 은유를 완벽히 구현한 고급 UI

초기에는 기능성과 명확성이 더 중요합니다.

---

## 단계별 범위

## Phase 1: 기초 MVP

집중 영역:

* 아키텍처 골격
* 에이전트 경계 정의
* 공유 메모리 기본 구조
* mock 또는 기본 읽기 연동
* 직접 에이전트 대화
* 기본 협업 프로토콜
* 기본 브리핑 로직

기대 결과:
핵심 컨셉을 검증할 수 있는 초기 사용 가능 시스템

## Phase 2: PM 실용성 강화

집중 영역:

* 더 좋은 요약 품질
* 더 나은 아이디어 관리
* 더 강한 리스크 노출
* 회의 전/후 보조
* 더 유용한 선제 개입
* confidence labeling 개선

기대 결과:
일상적인 PM 업무에서 “실제로 꽤 도움된다”는 느낌을 주는 단계

## Phase 3: 승인 기반 액션 레이어

집중 영역:

* task 생성 제안
* task 업데이트 제안
* 메시지 초안 작성
* 구조화된 follow-up 제안

기대 결과:
행동을 직접 수행하지 않되, 승인 기반으로 실행 보조까지 확장된 단계

## MVP 성공 기준

MVP는 사용자가 아래를 안정적으로 수행할 수 있으면 성공입니다.

1. 적절한 질문을 적절한 에이전트에게 할 수 있다
2. 여러 에이전트가 관여해도 하나의 일관된 답을 받는다
3. 사실 정보가 올바른 source system에 근거하고 있다고 신뢰한다
4. 선제 알림이 방해보다 도움이 된다고 느낀다
5. PM 업무 흐름 안에서 반복적으로 사용한다

---

## 초기 KPI 제안

### 사용성

* 일간 / 주간 활성 사용
* direct agent conversation 수
* multi-agent response 비율
* 에이전트별 engagement

### 품질

* 응답 유용성 체감
* “틀림 / 쓸모없음” 피드백 비율
* 선제 알림 open / dismiss 비율
* 일일 브리핑 신뢰도

### 업무 영향

* 회의 준비 시간 절감
* 놓친 마감 / unnoticed blocker 감소
* task recall 부담 감소
* 우선순위 판단 속도 개선

---

## MVP 설계 원칙

MVP는 아래 핵심을 검증하는 최소한의 유용한 동작 집합이어야 합니다.

* 눈에 보이는 역할 분리
* 구조화된 협업
* 강한 요약
* 안정적인 source-of-truth 처리
* 유용한 브리핑
* 보수적인 선제 발화

어떤 기능이 복잡도만 높이고 이 기반을 강화하지 못한다면, MVP에서는 미루는 것이 맞습니다.

## 핵심 사용자 시나리오 3개 (구체화)

아래 시나리오는 모두 MVP 범위 내(read-only 외부 조회, 구조화 협업, 공유 메모리 활용)에서만 동작합니다.

### 시나리오 1: 신규 기능 아이디어의 2주 내 착수 가능성 점검 (서비스 기획 중심)

1. 시나리오 이름
- 신규 기능 아이디어의 2주 내 착수 가능성 점검

2. 사용자 목표
- 새로 떠오른 기능 아이디어가 실제로 단기 착수 가능한지 전략/운영/일정 관점으로 빠르게 판단한다.

3. 시작 조건
- 사용자가 서비스 기획 Agent를 명시적으로 선택한 상태
- 최근 프로젝트 상태가 Asana에 존재
- 이번 주/다음 주 일정 정보가 Google Calendar에서 조회 가능

4. 사용자 입력 예시
- "온보딩 첫 화면에 산업군별 템플릿 추천 기능을 넣고 싶어. 2주 안에 시작 가능한지 봐줘."

5. 주 응답 에이전트
- 서비스 기획 Agent

6. 내부적으로 협업하는 에이전트
- 제품 운영 Agent: 현재 진행중 작업, 마감, blocker 기반 현실성 확인
- 개인 비서 Agent: 사용자의 실제 검토 가능 시간/회의 밀도 확인

7. 외부 도구 조회 목록
- Asana (read)
- Google Calendar (read)
- Slack (read, 관련 아이디어 스레드/피드백)
- Notion (read, 기존 기획 노트)
- Web Search (read, 유사 사례 빠른 참고)

8. 단계별 상호작용 흐름
- 사용자가 서비스 기획 Agent에 아이디어와 기한 기대를 전달한다.
- 서비스 기획 Agent가 아이디어를 핵심 가설/가치/선행조건으로 분해한다.
- 서비스 기획 Agent가 제품 운영 Agent에 `feasibility_check`를 보낸다.
- 제품 운영 Agent가 Asana 기반으로 현재 스프린트 부하, 임박 마감, blocker를 요약 반환한다.
- 서비스 기획 Agent가 개인 비서 Agent에 `schedule_check`를 보낸다.
- 개인 비서 Agent가 Calendar 기반으로 사용자의 검토 가능 시간창과 충돌 일정을 반환한다.
- 서비스 기획 Agent가 Slack/Notion/Web Search 맥락을 보강해 "착수 가능/조건부 가능/보류"로 정리한다.
- 서비스 기획 Agent가 근거와 불확실성을 함께 사용자에게 단일 응답으로 제공한다.

9. 최종 사용자에게 제공되는 결과
- 아이디어 1문장 정의
- 2주 내 착수 가능성 판정(confirmed/likely/tentative/needs_review)
- 착수 전 필수 선행조건 2~3개
- 일정/운영 리스크 요약
- 지금 바로 검토할 질문 2개

10. 저장/갱신되는 memory
- ideas: 아이디어 요약, 기대 효과, 관련 서비스 태그
- project summaries: 현재 용량/리스크 요약 스냅샷
- important context notes: "2주 착수 검토" 의사결정 근거

11. 실패 가능 지점 및 fallback
- Asana 데이터 지연/권한 오류: 최근 공유 메모리 기준으로 임시 판정 후 needs_review 라벨 부여
- Calendar 충돌 정보 불충분: 개인 비서 Agent가 "시간정보 제한"을 명시하고 보수적 일정 추정 제공
- Web Search 신뢰도 낮음: 외부 사례는 참고로만 표시하고 내부 데이터 기반 결론 우선 제시

### 시나리오 2: 이번 주 전달 리스크 브리핑 (제품 운영 중심)

1. 시나리오 이름
- 이번 주 전달 리스크 브리핑

2. 사용자 목표
- 이번 주 내 지연 가능성이 높은 프로젝트와 주요 원인을 빠르게 파악한다.

3. 시작 조건
- 사용자가 제품 운영 Agent를 선택
- Asana 프로젝트/태스크 상태가 조회 가능
- 관련 의사결정 맥락이 Notion/Slack에 일부 존재

4. 사용자 입력 예시
- "이번 주에 가장 위험한 프로젝트 3개만 근거랑 같이 알려줘."

5. 주 응답 에이전트
- 제품 운영 Agent

6. 내부적으로 협업하는 에이전트
- 서비스 기획 Agent: 각 프로젝트의 전략 중요도/의존 기능 맥락 보강
- 개인 비서 Agent: 사용자 본인이 개입 가능한 시간창 확인

7. 외부 도구 조회 목록
- Asana (read)
- Slack (read, blocker/의사소통 신호)
- Notion (read, 프로젝트 문서/결정 로그)
- Google Calendar (read)

8. 단계별 상호작용 흐름
- 사용자가 제품 운영 Agent에 주간 리스크 요약을 요청한다.
- 제품 운영 Agent가 Asana에서 마감 임박, overdue, blocker 태스크를 집계한다.
- 제품 운영 Agent가 Slack 신호(반복 이슈, 답변 지연, 긴급 키워드)로 운영 위험도를 보정한다.
- 제품 운영 Agent가 서비스 기획 Agent에 `priority_review` 요청을 보내 전략 중요도를 수신한다.
- 제품 운영 Agent가 개인 비서 Agent에 `user_attention_check` 요청을 보내 사용자의 개입 가능 시간창을 수신한다.
- 제품 운영 Agent가 위험도(영향 x 긴급도 x 대응가능성) 기준으로 상위 3개를 선정한다.
- 제품 운영 Agent가 각 리스크의 근거/불확실성/즉시 확인 포인트를 묶어 사용자에게 응답한다.

9. 최종 사용자에게 제공되는 결과
- 위험 프로젝트 Top 3
- 프로젝트별 위험 원인 1~2개
- 신뢰도 라벨(confirmed/likely/tentative)
- 사용자가 오늘 확인할 체크포인트 목록

10. 저장/갱신되는 memory
- project summaries: 주간 위험도 스냅샷, 핵심 blocker, 추세 메모
- important context notes: "이번 주 Top risk" 판단 근거
- personal task summaries: 사용자 개입 필요 항목 요약(읽기 기반)

11. 실패 가능 지점 및 fallback
- Asana 필드 누락/프로젝트 구조 불일치: 공통 최소 필드(마감/상태/담당) 기준으로 축소 분석
- Slack 신호 과다로 노이즈 증가: 반복/우선순위 높은 신호만 필터링해 반영
- 전략 중요도 합의 불가: 서비스 기획 관점은 별도 의견으로 분리하고 운영 사실 기반 순위 우선 제공

### 시나리오 3: 오늘 업무 시작 브리핑 (개인 비서 중심)

1. 시나리오 이름
- 오늘 업무 시작 브리핑

2. 사용자 목표
- 오늘 일정, 반드시 처리할 일, 주의가 필요한 신호를 5분 안에 파악한다.

3. 시작 조건
- 사용자가 개인 비서 Agent를 선택했거나, opt-in된 오전 브리핑 시점
- Google Calendar, Notion, Slack 읽기 접근 가능
- 프로젝트 요약 메모리가 최근 상태로 유지됨

4. 사용자 입력 예시
- "오늘 내가 뭐부터 해야 하는지 우선순위로 정리해줘."

5. 주 응답 에이전트
- 개인 비서 Agent

6. 내부적으로 협업하는 에이전트
- 제품 운영 Agent: 오늘 미루면 위험해지는 프로젝트 항목 확인
- 서비스 기획 Agent: 오늘 회의/검토 안건의 전략 맥락 한 줄 보강

7. 외부 도구 조회 목록
- Google Calendar (read)
- Notion (read)
- Slack (read, 멘션/긴급 신호)
- Asana (read, 오늘 관련 마감/overdue)

8. 단계별 상호작용 흐름
- 개인 비서 Agent가 Calendar에서 오늘 일정과 집중 가능 시간창을 계산한다.
- 개인 비서 Agent가 Notion/Asana에서 오늘 관련 개인 업무와 마감 항목을 수집한다.
- 개인 비서 Agent가 Slack 멘션 중 긴급/중요 신호를 추린다.
- 개인 비서 Agent가 제품 운영 Agent에 `risk_check`를 보내 "오늘 미루면 커지는 위험"을 확인한다.
- 개인 비서 Agent가 서비스 기획 Agent에 `context_summary_request`를 보내 주요 회의의 전략 맥락을 보강한다.
- 개인 비서 Agent가 지금/다음/나중 포맷으로 사용자 실행 순서를 정리한다.
- 개인 비서 Agent가 단일 브리핑 응답으로 제시하고, 불확실 항목은 needs_review로 표시한다.

9. 최종 사용자에게 제공되는 결과
- 오늘 일정 핵심 요약
- 지금/다음/나중 우선순위 목록
- 반드시 확인할 멘션/이슈
- 회의 전 10분 준비 포인트

10. 저장/갱신되는 memory
- personal task summaries: 오늘 우선순위 스냅샷
- important context notes: 주의가 필요한 일정/멘션 근거
- project summaries: 사용자 개입이 필요한 프로젝트 연결 메모

11. 실패 가능 지점 및 fallback
- Calendar/Notion 동기화 지연: 마지막 성공 조회 시점 명시 + 메모리 기반 임시 브리핑 제공
- Slack 긴급도 판정 불확실: 긴급 기준 미충족 항목은 정보성으로 분리
- 다수 도구 응답 지연: 핵심 3항목(오늘 일정/가장 위험한 일/첫 실행)만 축약 제공
